# design-extract — 品牌设计系统提炼

把零散的视觉偏好收敛成两个可执行文件：

- **design.md** — 人读的品牌 SSOT：命名调性、版式语法、配色 token、字体配对、视觉语言、Do/Don't
- **theme.css** — 机器执行的翻译：`:root` 变量 + 品牌元件样式，可直接被幻灯片/封面/视频工具消费

与用户**全程中文交流**。

## 定位与边界

一句话使命：**从用户身上提取一套品牌视觉语言，落成 design.md + theme.css。**

任何新需求（用户提的、执行中自己想到的）先过三问，任一不过就不做、明确告知超出范围：

1. 它是**视觉语言**的成分吗？（文案语气、内容选题 → 出界）
2. 它能**落进 design.md / theme.css** 吗？（落不进 → 出界）
3. 做的是**提取**，还是设计服务 / 模板生产？（后两者 → 出界）

非目标（明确不做）：内容策略（声线/标题句式/文案规则）、版式模板库（具体页面布局是消费方引擎的事，本 skill 只提取版式语法规则）、logo/头像设计（用户自备素材）、动画编排。

## 流程总览

```
① 模式判定 → ② 开场访谈(对话) → ③ 方向板轮(整板单选，定版式骨架)
→ ④ 细节轮循环(浏览器点选)
        ↑              │ 不满意区块重生成
        └──────────────┘ 满意区块锁定
→ ⑤ 收尾对话(暗色变体 + 调性回放 + 命名 + Do/Don't 确认) → ⑥ 产出 design.md + theme.css
```

视觉判断永远走 picker（hex 码读不出感觉），语言判断永远走对话（"工程感还是手作感"渲染不成色块）。不要用对话问"你喜欢 #F4F1E8 吗"这种无效问题。

对话也只出选择题，不出填空题——**用户不需要会表达**。人不擅长生成、擅长纠正：所有语言判断都给 2-3 个候选让用户挑和修正（AskUserQuestion 的 Other 是兜底，不是主路径），调性这类东西从用户的点选行为里读出来再回放确认，永远不让用户当起草人。

## ① 模式判定

输入是个光谱，按用户手上有什么定模式（可混用）：

| 模式 | 触发条件 | 候选从哪来 |
|------|---------|-----------|
| **素材提取** | 用户有自己的旧封面/作品截图/旧 CSS/banner | 分析素材：聚类高频色、识别字体气质、找反复出现的元件——候选 = 用户已有风格的整理与提纯 |
| **参考借鉴** | 用户能给出喜欢的账号/网站/海报链接或截图 | 分析参考：提取的是**气质和手法**（色彩关系、字体类别、质感方向），绝不照搬具体画面——候选 = 同气质的原创变体 |
| **零素材探索** | 什么都没有 | 加强版访谈 + 读 `references/style-archetypes.md` 原型库，把访谈答案映射到 2-3 个原型——候选 = 原型 + 按访谈微调的变体 |

开场先问一句"有没有已有素材或喜欢的参考"，没有再进探索模式。素材质量只影响第一轮候选命中率，不是必要条件——循环会补。

## ② 开场访谈

用 AskUserQuestion，2-3 个问题定方向，避免第一轮候选全军覆没：

- **内容领域和受众**：讲什么、讲给谁（讲代码给工程师 vs 讲理财给小白，气质完全不同）
- **气质词**：给 4 组左右预设气质组合（如"锐利·高能量·冲击力" / "极简·留白·高级感"）让用户挑 1-2 组——不要让用户自己造词，Other 兜底
- **反向题**（探索模式必问）：从原型库的气质族里挑对比强的 4-5 种问"哪种最讨厌"——否定比肯定好答，直接排除一族
- **类比题**（可选）："你的内容像一本什么杂志 / 一家什么店"

素材提取模式可精简为一个问题（素材已经回答了大部分）。

## ③ 方向板轮（先定骨）

隔三米认出一个人，靠的是体格和姿态，不是衣服颜色——品牌同理：**明度结构和构图骨架先于色彩字体**。所以第一轮不出零件，出整板。

读 `references/style-archetypes.md`（每个原型自带版式签名），按访谈/素材分析做 **3-4 个完整方向板**：每板是一张自带构图模板的完整实景小样（迷你封面），板与板之间是**骨级差异**——构图骨架、明度结构、留白密度都不同，不是同一版式换皮。配比同样七三：七成贴画像，三成试探边界。

用户**整板单选**（数据格式见 `references/picker-protocol.md`《方向板轮》一节）：

- 选中无反馈 → 锁定**版式语法**（明度结构/构图骨架/留白密度），该板模板成为细节轮的预览模板；板上的色彩字体质感只是初始值，细节轮继续打磨
- 选中 + feedback → 以该板为基按反馈改一版重出
- "都不是" + 理由 → 换骨重出新板，理由记入红线池

## ④ 细节轮循环（再上肉）

方向定骨后，逐区块打磨。候选共享定稿方向的预览模板、只差变量值，仍是**实景小样**——`#F4F1E8` 和 `#F2EFE6` 在表格里肉眼不可分，渲染成两张小封面立刻能选。**每个区块 2-4 个候选**（例外：色板与字体配对放宽到 6-10 个，按画像匹配度排序——组合空间大、缩略图一眼可比，不会选择瘫痪）。候选间须是方向级差异、按七三配比留试探项，并遵守原型库《混搭规则》：原型不是套餐，每个区块至少混入 1 个跨原型候选——个性来自组合，不来自单选。

需要候选的区块（核心五项必出，动效按需）：

| # | 区块 | 提取什么 | 落到 theme.css |
|---|------|---------|---------------|
| 1 | 色板 | 5-8 个 token：底色/次级底/主文字/结构色/点缀色/徽章色/次级灰，每色**绑定用途**，定稿时连同**用量配比** | `:root` 颜色变量 |
| 2 | 字体配对 | 标题/正文(中文)/等宽三个角色 + 字重 + 层级对比 | 字体变量 + `@import` |
| 3 | 质感底纹 | 纯色/方格纸/点阵/噪点/无 + 密度 | 背景纹理变量 |
| 4 | 标志性元件 | 印章/徽章/定位十字/角标/分隔线/强调手法（高亮/下划线/描边）等"签名" | 组件类（`.stamp` 等） |
| 5 | 形状语言 | 圆角 vs 直角、硬投影 vs 柔光、实线 vs 虚线 | 卡片/按钮结构参数 |
| 6 | 动效气质 | 利落快切 vs 从容缓动（第二梯队，按需） | 速度/缓动变量 |

排版密度与构图已在方向板轮定为版式语法，不重复出块；命名调性和 Do/Don't 不进 picker，由对话提取（见⑤）。

循环机制（详细协议读 `references/picker-protocol.md`）：

1. 在工作目录建 `design-extract-picker/`，写 `picker.html`（复制 `assets/picker/picker-shell.html`）+ `picker-data.js`（候选数据：每个候选带 CSS 变量集 + 预览模板）
2. 起服务器（**必须后台运行**）：`node <skill目录>/assets/picker/server.mjs --dir design-extract-picker --port 4399`，然后 `open http://localhost:4399`
3. 后台等 `choices.json` 出现（until-loop 轮询），用户点"确认提交"的瞬间即可读取
4. 读 choices.json，逐区块处理：
   - 直接选中/微调后选中 → **锁定**，下一轮不再出现
   - 选了候选但带 feedback → 以该候选为基础按反馈改一版，下轮只出该区块
   - "都不是" + 理由 → 换方向重出 2-3 个新候选，**把理由记入红线池**（喂给⑤的 Do/Don't）
5. 重复直到全部锁定。每轮只重做不满意的区块——三五轮内必然收敛，不许每轮推倒重来

反馈分三层，picker 页面都已支持：页面内微调控件（明度/字重/密度档位，消化"接近但差一口气"）→ 选最接近的+文字反馈 → "都不是"+必填理由。

## ⑤ 收尾对话

全部锁定后，四件事。统一原则：**用户是裁判不是起草人**——每一件都是"我提炼候选 → 用户挑/纠正"，不问开放题：

1. **暗色变体**（可选）：问一句"内容会出现在暗色场景吗（夜间模式/视频暗场）？"要的话按已锁色板推导 2-3 个暗化方向候选（推导纪律见 `references/design-md-template.md`），加开一轮单区块 picker 确认——暗色不是新设计，是定稿色板的同色相翻译，不许静默交付。若定稿主色板本身是暗色，则反向问亮色变体（白天/打印场景），推导纪律同理反转
2. **调性回放**：调性不是问出来的，是从循环行为里**读出来**的。盘点行为数据——锁定的候选偏向哪些原型、每次否决的理由原话、反馈里反复出现的词——提炼 2-3 版调性描述（气质公式 + 一段画面感描述），回放给用户："听下来你要的是这种感觉？"用户挑一版，再口头修正。绝不问"你的调性是什么"
3. **命名**：基于选定的调性版本起 2-3 个体系名（如 "Engineering Blueprint"），AskUserQuestion 让用户挑
4. **Do/Don't 确认**：把循环中攒下的红线池（每次"都不是"的理由、用户的否定反馈）整理成列表展示："这几条要写进 Do/Don't 吗？"用户勾选确认。红线不是问出来的，是沉淀出来的

## ⑥ 产出

读 `references/design-md-template.md` 按六段式写 design.md（版式语法段来自方向板轮的锁定结果）。关键纪律：

- **每个值可执行**：hex 码、字体名、具体元件，不写"米白色"这种散文
- design.md 是 SSOT，theme.css 只是它的影子——文件头注明"由 design.md 翻译，改品牌先改 design.md"

theme.css 翻译规则：

- 所有颜色定义在 `:root` token；需要透明度的色加 `--xxx-rgb: R,G,B` 伴生变量，用 `rgba(var(--xxx-rgb), .12)` 引用——**`:root` 之外零硬编码色值**
- 只写"品牌长什么样"（token + 元件外观），不写布局/动画结构（那是消费方引擎的事）
- 暗色变体（若有）写进 `[data-theme="dark"]` 块：与 `:root` 同名 token 全量重声明，消费方加一个属性即整体换肤
- 第三方官方色（平台按钮等）不属于品牌主题，不放 theme.css
- 字体用 Google Fonts CDN `@import`（离线 woff2 内嵌是消费方的后续优化）

产出后把两个文件路径报给用户，建议浏览器打开最终预览页让用户终验。

## 清理

结束时杀掉 picker 服务器（`pkill -f server.mjs` 或记录的 PID），`design-extract-picker/` 临时目录问用户是否保留（里面有定稿预览，可能想留）。
